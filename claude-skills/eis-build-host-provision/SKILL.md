# EIS build/deploy host provisioning + EKS access

## What it is
EIS client accounts put **all build/CI/deploy EC2 hosts in the shared `infra` stage** (`projects/aws/<client>/terraform/lower/infra/services/`), NOT in the target stage's VPC. `lower/<stage>/services` has eks/rds/msk/s3 but **no ec2 module**. A "build host for UAT/<stage>" therefore = a new instance in `infra/services`, reaching the target cluster over the existing TGW path. Hosts are SSM-managed RHEL; deploy tooling typically runs in pipeline containers, so the host itself usually needs only base RHEL + SSM + (maybe docker) + an EKS access entry ‚Äî confirm scope with the requester (infra/access vs full ansible config).

Reference impl (live 2026-06-15, COEXT-105506, CAA MR !76): dedicated UAT deploy host `aws0caaudep01` cloned from the shared `bld01` config, UAT-scoped IAM, cluster-admin on `aws0caatesteks01`.

## Workflow
1. **Check if a shared host already serves the target.** The existing `bld01`/`jnk01` group IAM often already covers all clusters (`<prefix>eks*`/`deveks*`/`testeks*`) and the cluster's `access_mapping` may already map `ec2_group=bld`. Verify live: `aws ec2 describe-instances --filters Name=tag:Name,Values=<prefix>bld01`, `aws eks list-access-entries --cluster-name <cluster>`. If a **dedicated** host is still wanted (manager's call), continue.

2. **Pick a collision-safe acronym** (see Gotcha 1). NOT `ubld`/`uatbld` (contains `bld`). Use e.g. `udep` (UAT deploy). Instance key `<acronym>01` ‚Üí `<prefix><acronym>01`, role `<prefix><acronym>01-Role`, group policy `<prefix><acronym>-Policy`.

3. **Terraform `lower/infra/services/`** (mirror the CyberArk/CUSTOM-marker convention ‚Äî `ec2_settings` is a template-owned var, edit inline with `# BEGIN/END CUSTOM <id> | <JIRA>` blocks):
   - `variables.tf` `ec2_settings.<acronym>` ‚Äî clone `bld` (enable_policy, m6a.xlarge, `root_block_device {volume_size=300, volume_type="gp3", encrypted=true}`, `custom_rules=["30000-30010/tcp"]`, `additional_prefix_lists=["workspaces-dev"]`, Backup tag). **`encrypted=true` is safe on a fresh host** (no existing volume to flip false‚Üítrue ‚Üí no replacement).
   - `terraform.tfvars` `ec2` map: `<acronym>01 = {}` (all behaviour merges from settings).
   - NEW `files/iam/ec2/<acronym>.json` ‚Äî **stage-scoped** least-privilege: EKS + Secrets limited to `<prefix><stage>eks*` (e.g. `testeks*`) only (drop the all-clusters breadth of `bld.json`), ECR push/pull, S3 `<prefix>*portal-*`. Rendered via templatefile vars `project_prefix/account_id/region`.

4. **Terraform `lower/<stage>/services/terraform.tfvars`** ‚Äî add a NEW key to `eks["01"].access_mapping` (non-destructive; leave existing `ci`/`ci-etcs` alone):
   ```hcl
   "ci-uat" = { ec2_group = "<acronym>", policy = { arn = "arn:aws:eks::aws:cluster-access-policy/AmazonEKSClusterAdminPolicy", scope = "cluster" } }
   ```

5. **Ordered apply** (Gotcha 2): `atlantis apply -p lower-infra-services` ‚Üí re-`atlantis plan -p lower-<stage>-services` (now shows `+ aws_eks_access_entry` + `_policy_association` for the new role) ‚Üí `atlantis apply -p lower-<stage>-services` ‚Üí merge. A plain ordered `atlantis apply` works because exec-order puts infra (13) before stage (33); a targeted stage-first apply silently skips the access entry.

6. **Ansible host-config** (`projects/aws/<client>/ansible`) ‚Äî group_vars/`<acronym>`.yaml + add acronym to `playbooks/linux_joindomain.yaml` `hosts:`. `environment`/`docker_install`/`vault_agent` are `hosts: all` so `-l <host>` picks the new host up. **Often deferred / SSM-only** ‚Äî confirm whether the host needs domain-join + docker, or if pipeline-container tooling suffices.

## Gotchas
1. **eis-eks `ec2_group` is a SUBSTRING matcher.** `eis-eks/data.tf`: `data.aws_iam_roles.ec2_group` uses `name_regex = ".*${ec2_group}.*"`. So an acronym containing an existing `ec2_group` value (`bld`, `jnk`) double-binds: e.g. `ci={ec2_group:"bld"}` would resolve BOTH `bld01-Role` and `ubld01-Role` ‚Üí EKS rejects two access entries for one principal ‚Üí **apply fails**. Grep the chosen acronym vs every existing `ec2_group` both ways before committing.
2. **Empty-lookup ordering trap.** The `access_mapping` `ec2_group` lookup runs at PLAN time. If `lower/<stage>/services` plans before `<acronym>01-Role` exists, the access entry is **silently not created** (no error) ‚Äî you'd need a second apply. Always create the host (infra) first, then re-plan the stage.
3. **CI pipeline red with green pre-commit** = the GitLab pipeline's `external` stage carries Atlantis commit statuses (invisible to the REST jobs API). See atlantis-debug skill / Gotcha in that skill.

## E2E verification (SSM, no ansible needed)
Prove the host authenticates to the cluster as the access entry grants. SSM `AWS-RunShellScript` on the instance:
- Install tools: `dnf install -y unzip`; aws-cli v2 (`awscli.amazonaws.com/awscli-exe-linux-x86_64.zip` ‚Üí `./aws/install --update`); kubectl (`dl.k8s.io/release/v<clusterver>.0/bin/linux/amd64/kubectl`). Egress for both is usually open via TGW (`eks.<region>` 403 = reachable).
- **Trap: SSM root shell `$HOME` ‚âÝ `/root`** ‚Üí kubectl hits `localhost:8080`. Export `KUBECONFIG=/root/.kube/config` (and `HOME=/root`) explicitly after `aws eks update-kubeconfig --kubeconfig /root/.kube/config`.
- Proof commands: `aws sts get-caller-identity` (= `<acronym>01-Role`); `kubectl auth can-i '*' '*'` ‚Üí yes; `kubectl get ns` (target ns Active); `kubectl get nodes`; `kubectl auth can-i create deployment -n <target-ns>` ‚Üí yes; `kubectl create configmap x --from-literal=a=b -n <target-ns> --dry-run=server -o name` (real write authz, not persisted); `kubectl auth whoami` ‚Üí `Extra.sessionName = <instance-id>` (cluster sees the host).
- **Cleanup after test** (if ansible/host-config is deferred): `rm -rf /root/.kube /usr/local/aws-cli /usr/local/bin/aws /usr/local/bin/aws_completer /usr/local/bin/kubectl` to leave the host pristine for the team's own config.
- No-host network-only proof: `curl -sk <api-endpoint>/version` from the host ‚Üí HTTP 401 = TLS + reachable via TGW + control-plane SG (which admits the `local` prefix, 10/8 ‚äá infra subnets).

## Tooling persistence ‚Äî DON'T over-clean a host people will actually use (COEXT-105506, 2026-06-17)
The E2E "cleanup after test" above is **only** correct when the team will run their own config that installs deploy tooling. A bare deploy host ‚Äî aws-cli/kubectl removed, nothing in the ansible `bld` config installs them ‚Äî leaves real users hitting **`aws: command not found`** the first time they SSM in (vfiasko, COEXT-105506). The bld ansible role does NOT install aws-cli/kubectl. So:
- If the host is a deploy host people log into and run `aws eks update-kubeconfig`/`kubectl`, **keep the tools** ‚Äî install aws-cli v2 + kubectl **system-wide to `/usr/local/bin`** (root, via SSM) so every login user inherits them. Durable answer = add both to the bld ansible role (idempotent), else a manual `/usr/local/bin` install dies on host replacement.
- **Stale-shell trap:** if you install the binaries *after* a user already opened a shell, their pre-install login session caches the negative lookup ‚Üí still `command not found`. Fix in that shell: `hash -r` or reconnect. `/usr/local/bin` is on the RHEL default login PATH (`/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin`) for **all** users identically ‚Äî a per-user difference is almost always one user on a stale shell, not a PATH/permission issue. Verify with `su - <user> -c 'echo $PATH; command -v aws'`.
- **Shared instance-role creds:** the host's EKS access comes from the EC2 instance role via IMDS, so **any** user who can log in inherits the same cluster access (cluster-admin here) with no `--profile`. Expected for a shared build/deploy host ‚Äî if per-user isolation is required, that's a different design (don't grant cluster-admin to the instance role).

## Variant: host INSIDE the stage VPC, test/services-owned (COEXT-105506, RESOLVED 2026-06-16)
When the host must live in the **stage's own VPC** (UAT isolation), not shared infra:
- Define it in `lower/<stage>/services` via the **fleet ec2-map** (`ec2.tf` + `variables_custom.tf` + `terraform.tfvars` ec2 map), which the CyberArk migration (COEXT-105281) added to test/services. data.tf already has core VPC/subnets/route53/prefix_lists there.
- **Naming**: the stage `project_prefix` includes the stage (e.g. `aws0caatest`), so key `bld01` ‚Üí `aws0caatestbld01` (the FIRST build host *in that stage*; `bld02` only applies in infra where `bld01` is taken). Build nodes are all `bld` (per Markuss). IMPORTANT: `${project_prefix}eks*` in the stage = `aws0caatesteks*` (NOT `${project_prefix}testeks*`).
- **KEY-COLLISION (critical):** eis-eks builds the access-entry key as `regex("${ec2_group}[0-9]+", arn)`. `ci={ec2_group="bld"}` resolves BOTH the new `aws0caatestbld01` and the shared infra `aws0caabld01` ‚Üí both ‚Üí key `bld01` ‚Üí duplicate-key plan FAIL. Fix: set the stage's `ci` mapping `ec2_group="testbld"` (resolves only the in-stage host, key `ci-testbld01`; drops shared infra bld01 from the stage cluster = isolation). If the host is already live when you hit this, switching ci‚Üítestbld is just an access-entry swap (no host churn).
- **IAM**: a GROUP `bld` policy (`enable_policy=true`) renders the stray test `files/iam/ec2/bld.json` which refs `${dns_zone}` (unsupplied by the group templatefile ‚Üí tflint/plan fail). Use `custom_policy=true` on the ec2-map entry + a per-instance `files/iam/ec2/bld01.json` ‚Üí `aws0caatestbld01-Policy` (UAT-scoped to `${project_prefix}eks*`).
- **DNS name** of the in-stage host = `<key>.<stage_private_zone>`. The eis-ec2 ec2-map sets `route53_zone_name = data.aws_route53_zone.stage_private.name`, and the test core creates the stage private zone `<stage>.aws<region>.<domain>` (`create_stage_zone`). So the CAA UAT build host resolves at **`aws0caatestbld01.test.aws0.caa-eis.cloud`** (A ‚Üí private IP, e.g. 10.34.88.59) ‚Äî NOT the infra zone. The OLD shared `aws0caabld01.infra.aws0.caa-eis.cloud` is a *different* host whose access to the stage cluster was deliberately dropped (the `testbld` matcher excludes it) ‚Üí it now 401s on the UAT cluster. When someone asks "what's the DNS name of the build host", give the in-stage FQDN and explain the infra host is no longer authorized.

## Ansible bld-node config (Docker runner) ‚Äî gotchas
Run `projects/aws/<client>/ansible/docker/run.sh playbook playbooks/<pb>.yaml --limit <host> --become` with `VAULT_TOKEN=$(cat ~/.vault-token)` (validate via curl `‚Ä¶/v1/auth/token/lookup-self`) + `AWS_PROFILE`. Playbooks for a bld node: `environment`, `docker_install`, `linux_joindomain`. **`vault_agent_ssl` is N/A for build hosts** (no TLS endpoint; shared bld01 has vault-agent inactive). The host is Acronym=`bld` ‚Üí uses existing `group_vars/bld.yaml` + `linux_joindomain hosts: bld,jnk` (no new group_vars/host needed).
- **Image gaps** (add to `ansible/docker/Dockerfile`): `passlib` (else `environment` user-hash task fails "passlib must be installed") and `community.docker` (else `docker_install` errors `community.docker.docker_login` unresolved). This file IS tracked in the consuming ansible repo ‚Üí normal MR there.
- **Order**: run `linux_joindomain` (AD join) BEFORE `docker_install` ‚Äî `docker_install` adds the AD user `caa_build` to the docker group, which requires it to resolve. The ansible `user` module CAN'T add an AD-only user to a LOCAL group (usermod needs /etc/passwd). Quick fix: `gpasswd -a caa_build docker`. **Durable role fix** (`docker_install/tasks/users.yml`): `getent group docker` + `gpasswd -a {{item}} docker` loop over `docker_users` with idempotent check `item not in getent_group.docker[2]|split(',')`.
- **Roles are VENDORED** ‚Äî `roles/.gitignore`=`/*` (only `local*`+requirements.yml tracked); each role pulled from `cv-devops/ansible/ansible-roles/<role>.git` via `roles/requirements.yml` (pinned, e.g. docker_install v1.1.0). Editing the vendored copy is ephemeral. To persist a role fix you're Reporter-only on cv-devops ‚Üí mirror to `iac/ansible/roles/<role>` (Maintainer), tag the next patch, bump the requirements.yml pin (vault_agent_ssl v1.1.1 / COEXT-104228 precedent). **Test the SHIPPED path (not the local edit):** `rm -rf roles/<role>` then `./docker/run.sh exec ansible-galaxy install -r roles/requirements.yml -p roles/` ‚Üí confirm `roles/<role>/meta/.galaxy_install_info` shows the new tag (proves the mirror+pin resolve), then run the playbook clean-room (e.g. `gpasswd -d <user> <grp>` first) and verify the pristine role does the fix.
- Verify: `systemctl is-active docker sssd`, `getent group docker` (incl. caa_build), `ls /etc/sudoers.d | grep <grp>_<client>_ssh`, `docker run --rm hello-world` PASS, registry login in `/root/.docker/config.json`.
